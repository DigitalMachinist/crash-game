# Crash Game MVP Review — 2026-03-11

## Executive Summary

The MVP implements a cryptographically fair crash game with provably fair verification via drand and hash chains. The core game logic is mathematically sound and multiplier calculations are correct. However, the system has **13 critical issues** across security, compliance, error handling, and state persistence that must be addressed before any real-money or production deployment.

The most severe issues involve unverified drand signatures (allowing cryptographic attacks), client-side balance storage with trivial manipulation, missing server-side state persistence for player joins/cashouts, lack of error handling in the Durable Object lifecycle, and complete absence of responsible gambling controls. Additionally, 14 high-priority issues cover DoS vectors, performance O(n²) bugs, and UX blind spots that leave players confused about transaction state.

**Recommendation:** Do not deploy to production with real money until all Critical and High findings are resolved.

## Critical Findings (must fix before production)

### Security

#### [Security-1] Drand beacon signatures not verified (cryptographic attack vector)
**File:** `src/server/drand.ts:62-90`, `src/client/lib/verify.ts`
**Issue:** The `DrandBeacon.signature` field is defined but never validated. A compromised network layer, man-in-the-middle attack, or a drand service returning forged randomness would propagate undetected to the crash point calculation. The signature exists in the response but is completely unused.
**Recommendation:** Import drand's BLS signature verification library; verify the beacon signature against drand's public key before accepting any beacon data; fail the round with an error if verification fails.

#### [Security-2] Client can spoof playerId to claim another player's pending payout
**File:** `src/server/crash-game.ts:113-123`
**Issue:** Pending payouts are stored in a `Map<playerId, payout>` keyed only by `playerId` without verifying WebSocket connection identity. An attacker can (a) observe a victim's UUID in localStorage, (b) wait for the victim to disconnect, (c) connect with the victim's UUID, and (d) claim the pending payout intended for the original connection.
**Recommendation:** Bind pending payouts to both `playerId` AND WebSocket connection ID; verify that a reconnecting player matches the original connection's identity via a token or similar mechanism.

#### [Security-3] No validation on autoCashout value
**File:** `src/server/game-state.ts:74-149`
**Issue:** Server accepts `autoCashout` values that are negative, NaN, or less than the starting multiplier (1.0). A player with `autoCashout: 0.5` will auto-cashout immediately at 1.0x since the multiplier starts at 1.0, leading to unexpected payouts or silent errors.
**Recommendation:** Add validation in `handleJoin()`: `if (msg.autoCashout !== null && (!Number.isFinite(msg.autoCashout) || msg.autoCashout < 1.0)) { return error; }`.

#### [Security-4] No length validation on playerId (DoS via memory exhaustion)
**File:** `src/server/crash-game.ts:125-143`
**Issue:** Accepts playerId values of 1000+ characters, stored in the player Map, serialized in state snapshots, and persisted to durable storage. Repeated joins with massive IDs exhaust memory and increase serialization overhead.
**Recommendation:** Add validation: `if (msg.playerId.length > 256) { return error; }`.

#### [Security-5] No rate limiting on join/cashout messages (DoS vector)
**File:** `src/server/crash-game.ts:102-168`
**Issue:** A malicious client can spam `join` or `cashout` messages, or rapidly reconnect to trigger expensive state snapshots and broadcasts. With 1000 concurrent connections sending 10 join messages/sec = 10,000 joins/sec with no backpressure.
**Recommendation:** Implement per-connection or per-playerId rate limiting; reject messages exceeding a threshold (e.g., 5 joins per 10 seconds).

### Compliance & Responsible Gambling

#### [Compliance-1] Client-side balance storage with no server authority (trivial manipulation)
**File:** `src/client/lib/balance.ts`
**Issue:** Balance is stored in plain-text localStorage; a player can trivially execute `localStorage.setItem('crashBalance', '999999')` to manipulate their balance. No audit trail, no server-side enforcement, no recovery path. This violates fundamental gambling compliance requirements.
**Recommendation:** Implement server-side balance authority; track all transactions server-side; return current balance in every message from the server; do NOT trust client localStorage for real-money operations.

#### [Compliance-2] No responsible gambling features (critical for any real-money product)
**File:** `src/App.svelte`, entire codebase
**Issue:** Completely missing: session time limits, deposit limits, loss limits, self-exclusion, cool-down periods, betting limits, or warnings. A player can bet continuously for hours with no protections or notifications.
**Recommendation:** Implement a responsible gambling suite: session time limit (e.g., 2 hours), deposit limit (e.g., $100/day), loss limit, warning notifications at thresholds, self-exclusion functionality, and cool-down periods.

### Backend / Durable Objects

#### [Backend-1] Player join state not persisted (data loss on DO eviction)
**File:** `src/server/crash-game.ts:125-143`
**Issue:** When a player calls `handleJoin()`, the player is added to the `players` Map, but `persistState()` is NOT called. If the DO is evicted immediately afterward (e.g., under memory pressure), the join is lost and the player cannot reconnect to their bet.
**Recommendation:** Call `persistState()` immediately after `handleJoin()` returns, before sending confirmation to the client.

#### [Backend-2] Player cashout state not persisted (data loss on DO eviction)
**File:** `src/server/crash-game.ts:157-167`
**Issue:** Similarly, `handleCashout()` updates the player's `cashedOut` flag but does NOT persist state. If the DO crashes between cashout and the next alarm tick, the cashout is lost and the player's balance appears to revert.
**Recommendation:** Call `persistState()` immediately after `handleCashout()` returns.

#### [Backend-3] Durable Object initialization has no error handling (uncaught exceptions during startup)
**File:** `src/server/crash-game.ts:62-85`
**Issue:** `onStart()` calls `generateRootSeed()`, `computeTerminalHash()`, and storage operations with no try/catch. If any initialization step throws, the DO crashes with no recovery path or graceful fallback.
**Recommendation:** Wrap all initialization in try/catch; log errors; implement a graceful fallback (e.g., initialize with a default seed) so the game loop can continue.

#### [Backend-4] Durable Object game loop has no error handling (frozen multiplier on crash)
**File:** `src/server/crash-game.ts:194-238`
**Issue:** `onAlarm()` calls phase handlers (`handleTick()`, `startRound()`, etc.) with no try/catch. If any handler throws during the RUNNING phase, no exception is caught, the alarm is not rescheduled, the game loop stops silently, and players see a frozen multiplier with no error message.
**Recommendation:** Wrap all game loop handlers in try/catch; always reschedule the alarm even on error; dispatch an error message to all clients; log the exception with full context.

#### [Backend-5] No error logging or observability (impossible to debug production issues)
**File:** `src/server/crash-game.ts` (entire file)
**Issue:** Zero console logging or structured logging anywhere in the server code. Failed drand fetches, storage errors, malformed messages, and exceptions have no visibility. Production debugging is impossible.
**Recommendation:** Add `console.error()` and `console.log()` at minimum; consider Cloudflare Logpush for structured logging in production; log every error with full context (message type, player ID, error details).

### Frontend / UX

#### [UX-2] Bet form disappears without confirmation when round starts

**Issue:** When the game phase transitions from WAITING to STARTING, the bet form disappears from the UI without any confirmation message. Players don't know if their bet was accepted, rejected, or lost in a network race condition.
**Recommendation:** Show a "Bet submitted" or "Round in progress" confirmation message before the form vanishes; keep the confirmation visible until the round crashes.

#### [UX-3] Client-side balance validation missing (confusing error messages)
**File:** `src/client/BetForm.svelte:31-33`
**Issue:** The bet form does not validate that the wager amount is ≤ current balance. Players can attempt to bet more than they have, get a server error, and be confused about why their bet failed.

## High Priority Findings (fix soon)

### Security & Validation

#### [High-1] No type validation on JSON messages (bypasses safety)
**File:** `src/server/crash-game.ts:105`, `src/client/socket.ts:36`
**Issue:** Server casts `JSON.parse(message) as ClientMessage` and client casts similarly without runtime type checking. Malformed messages with wrong field types (e.g., `wager: "abc"`) pass through without validation, leading to silent failures or cryptic errors.
**Recommendation:** Create a type guard function using `typeof` checks or a validation library (e.g., zod, valibot) to validate message shape and field types before processing.

#### [High-2] No validation of drand beacon structure (cryptic failures on malformed data)
**File:** `src/server/drand.ts:75`
**Issue:** After fetching from drand, the beacon is cast `as DrandBeacon` without validating required fields exist (`round`, `randomness`, `signature`). Missing fields propagate silently to `hexToBytes()` causing cryptic failures downstream.
**Recommendation:** Validate beacon structure before use: ensure `round` is a number, `randomness` and `signature` are strings; fail the round with a clear error if validation fails.

#### [High-3] Storage fields loaded without validation (corrupted state recovery)
**File:** `src/server/crash-game.ts:64-70`
**Issue:** Fields loaded from durable storage (`rootSeed`, `gameNumber`, `chainIndex`, etc.) are used without validation. If storage is corrupted or contains `undefined` values, the game proceeds with broken state and crashes unexpectedly.
**Recommendation:** Validate all required fields after loading from storage; if any required field is missing or invalid, log a warning and reinitialize from scratch.

#### [High-4] Server JSON cast provides zero runtime safety for client messages
**File:** `src/server/crash-game.ts:105`
**Issue:** Generic `JSON.parse(message) as ClientMessage` bypasses TypeScript safety. Invalid field types, missing required fields, and extra fields all pass through unchecked.
**Recommendation:** Create a discriminating type guard that checks `type` field presence and validity; validate wager/playerId/autoCashout field types; reject malformed messages with an error response.

### Backend / Performance

#### [High-5] Player join is not idempotent (poor reconnection resilience)
**File:** `src/server/game-state.ts:105-116`
**Issue:** If a player's first join succeeded but the server's confirmation message was lost before reconnect, the retry returns an error ("already joined") instead of success. This breaks reconnection resilience and creates a UX issue where players think their retry failed.
**Recommendation:** Return success if the player is already joined with the same playerId and wager; update the join timestamp silently to indicate reconnection.

#### [High-6] Join can be rejected due to race during WAITING→STARTING transition
**File:** `src/server/crash-game.ts:207-214`, `src/server/game-state.ts:79-90`
**Issue:** A join message sent during WAITING is blocked by `blockConcurrencyWhile`, queued, then rejected when processed during STARTING. The join was legitimate (sent while joining was allowed) but fails due to timing. Players experience an inconsistent "join window."
**Recommendation:** Document this as a known limitation, or implement a grace period (e.g., accept joins during the first 100ms of STARTING) to absorb network latency.

#### [High-7] O(n²) iteration during crash and auto-cashout (severe performance regression at scale)
**File:** `src/server/game-state.ts:268`, `src/server/crash-game.ts:335`, `src/server/game-state.ts:254-304`
**Issue:** During crash, the code iterates all players and calls `getConnections().find()` for each player (O(n²)). Similarly, auto-cashout checking iterates all 100+ players every 100ms. With 100 players: 10,000 comparisons per crash and 1,000 iterations/second during RUNNING. At 5000 players this becomes unusable.
**Recommendation:** Build a `Set<connectionId>` of active connection IDs once per phase; check membership in O(1). For auto-cashout, pre-sort players by threshold or reduce tick frequency to 200ms.

#### [High-8] buildStateSnapshot() is O(n) and called on every phase transition (broadcast overhead)
**File:** `src/server/crash-game.ts:97-99`, `src/server/crash-game.ts:297-299`
**Issue:** Full player list serialized on every phase transition broadcast (4 transitions/round × n players). With 100 players this is expensive; at 5000 players it becomes a bottleneck.
**Recommendation:** Implement delta-based broadcasting or cache snapshots between transitions; only send full state on connect and phase changes, and send deltas for player joins/cashouts.

#### [High-9] playersList derived store creates new array on every update (unnecessary re-renders)
**File:** `src/client/stores.ts:28`
**Issue:** `playersList` creates a new array `Object.values($players)` on every `players` store update, including every cashout. `PlayerList.svelte` re-renders the entire table even when only one player cashed out.
**Recommendation:** Memoize `playersList` to only recompute when player count or player identities change; use a comparison function to detect meaningful changes.

#### [High-10] Pending payouts Map is unbounded (memory accumulation over time)
**File:** `src/server/crash-game.ts:59`
**Issue:** The `pendingPayouts` Map accumulates entries indefinitely. Over many rounds with high player churn, could grow to 1+ MB. No cleanup mechanism.
**Recommendation:** Implement FIFO eviction (e.g., keep only last 100 rounds of payouts) or a round-based cap.

#### [High-11] No max players per round (unbounded memory growth and broadcast overhead)
**File:** `src/server/game-state.ts:74-149`
**Issue:** Server accepts unlimited joins, growing the player Map and broadcast size indefinitely. Affects memory, serialization time, and payout computation time.
**Recommendation:** Add a configured cap (e.g., 5000 players) and reject excess joins with a "room full" error message.

### Code Quality & Type Safety

#### [High-12] House edge hardcoded as literal '99' (sync risk with config.ts)
**File:** `src/client/lib/verify.ts:68`, `src/config.ts:22`
**Issue:** Config defines `HOUSE_EDGE = 0.01` (1%), but verify.ts uses a hardcoded literal `99` representing `(1 - 0.01) * 100`. No compile-time link between them. If config changes and verify.ts is not updated, client verification silently computes wrong crash points.
**Recommendation:** Import `HOUSE_EDGE` from config.ts into verify.ts; add a build-time test asserting `(1 - HOUSE_EDGE) * 100 === 99`.

#### [High-13] Phase handling has no exhaustiveness check (new phases won't cause compile error)
**File:** `src/server/crash-game.ts:194-237`
**Issue:** `onAlarm()` uses if/else chain for phase matching. Adding a new Phase value to the union won't cause a TypeScript error; the new phase will silently fall through.
**Recommendation:** Add `const _exhaustive: never = phase;` in the final else clause, or convert to a switch with `default: throw new Error(...)`.

#### [High-14] hexToBytes uses float division without integer coercion (byte corruption)
**File:** `src/server/drand.ts:95`, `src/client/lib/verify.ts:18`
**Issue:** Both functions use `bytes[i / 2]` where `i` is odd; JavaScript produces a float (e.g., `1/2 = 0.5`), which is coerced to index 0, causing incorrect byte assignments for odd indices.
**Recommendation:** Use `bytes[Math.floor(i / 2)]` or `bytes[(i / 2) | 0]` to explicitly convert to integer.

#### [High-15] Targeting routed error messages to specific players is broken
**File:** `src/server/crash-game.ts:137-166`
**Issue:** `OutboundMessage.targetPlayerId` is defined but the routing logic is ignored. Messages marked `broadcast: false` are sent to the current connection instead of finding the player by `targetPlayerId`.
**Recommendation:** Implement proper routing: if `broadcast === true` use `this.broadcast()`, else build a connection lookup table and route to the correct player's connection.
